# Sloth 0.3.0

_Unreleased_

Sloth 0.3.0 **slims the DI container** down to the surface services actually use,
and adds two capabilities: **`DI.lazy`** (break circular dependencies) and
**`warmup()`** (fail fast at boot). For the vast majority of services the upgrade
is **just a version bump** — the kept API (`DI.inject`, `DI.Type`, `DI.Scope`,
`container.register`/`resolve`) is unchanged in shape.

## ✨ Highlights

### `DI.lazy(token)` — break circular dependencies

`inject` resolves eagerly during construction, so two services depending on each
other throw `circular dependency detected`. `lazy` returns a transparent proxy
and defers resolution to **first use**, so the owner finishes constructing before
the cycle closes. Swap `inject` → `lazy` on **one edge** of the cycle — call
sites don't change:

```ts
class A {
  constructor(private readonly b = DI.lazy(Types.B)) {} // breaks the A <-> B cycle
  run() { return this.b.help() }                         // B resolved here, on first use
}
class B {
  constructor(private readonly a = DI.inject(Types.A)) {}
}
```

Methods are bound to the real instance (private `#fields` and fluent `return
this` work), and it removes registration-order sensitivity — the token only needs
to be registered before it is first *used*.

### `warmup()` — fail fast at boot

Because `inject`/`lazy` resolve on construction/first-use, a missing or
misconfigured dependency can surface on a request rather than at startup.
`warmup()` restores fail-fast: it eagerly resolves every registered class/factory
token once, validating the whole graph (and pre-building singletons) before you
serve traffic.

```ts
await app.start({ port: 3000, warmup: true }) // validate the graph, then listen
// or call it manually once everything is registered:
app.warmup() // { resolved: <count> }  — throws Errors.InitError listing failures
```

Singletons are constructed and cached; transients validated and discarded; value
providers skipped. Safe alongside `lazy`. Opt-in — `start()` does not warm up
unless asked.

### A much smaller container

The DI module (originally a di-wise fork) dropped from **18 files / ~1,130 lines
to 6 files / ~400**, keeping only what Sloth and its services actually used:
`Type` tokens, `Scope`, `register`/`resolve`/`has`/`warmup`, `inject`, and
`lazy`. Resolution is now a synchronous core (ambient current-container + a `Set`
for circular detection) — no behavior change for the kept API.

## ⚠️ Breaking changes

### Default scope is now `Singleton`

`container.register(token, { useClass })` **without** an explicit `{ scope }` now
defaults to **`Singleton`** (was `Transient`), aligning bare `register` with
`Controller.add`, `@Route`, and `ServiceBuilder.addClass`. A repo/service
registered in `inits/*` without a scope is now a single shared instance.

This is safe for the standard pattern: anything injected into a `Singleton`
handler is *already* shared across concurrent requests, so repos/services already
had to be stateless. For the rare dep that needs a fresh instance per resolve,
opt back in with `{ scope: DI.Scope.Transient }`.

### Removed (unused) DI surface

None of these were used by any service or by the framework itself:

| Removed | Replacement |
|---|---|
| `@Injectable` / `@Scoped` / `@AutoRegister` / `@Inject` / `@InjectAll` | `container.register(token, { useClass }, { scope })` |
| `injectAll` / `InjectAll` / `container.resolveAll` | resolve a single token |
| `injectBy` / `inject.by` / `Injector` token | `DI.lazy(token)` |
| multi-token `inject(a, b, …)` | inject one concrete token |
| `container.createChild()` (child/parent containers) | `createContainer()` |
| container middleware (`applyMiddleware`, `resolveAllSafe`) | — |
| `Build(...)` / `Value(...)` builder tokens | `{ useFactory }` / `{ useValue }` |
| `Type.inter` / `Type.union` | separate tokens |
